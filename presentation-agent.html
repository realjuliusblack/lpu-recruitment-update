What the Presentation Agent actually does.

One agent, channel-aware. Direct clients get an email. VMS clients get a portal submission on the next build. One state machine for the whole thing.

When a physician clears Qualification: the Presentation Agent looks up the client on file and decides how to present. For email-route clients, a short name clear email goes out with name, specialty, NPI, and availability. No CV, no rate on this first touch. Just a yes/no ask.

When the client replies: Claude reads the reply and classifies it yes, no, or ambiguous. Clear yes advances the state to presented so the full packet follows. Clear no flags the physician for rematch (Agent 2 picks them up for other roles). Anything unclear holds state and goes to Tyson with yes / no / needs-info buttons.

When the client goes silent: a polite follow-up lands at 48 hours. A second at 96 hours. Past that, Tyson gets a direct alert so he can intervene. No spam loops; every action fires once per application.

Every send is on a safety net: during MVP, every draft routes to Julius first so we can tune tone with Tyson before the first real facility sees anything. The client's actual contact is stored and ready; the flip is one switch once Tyson is walked through approvals.
